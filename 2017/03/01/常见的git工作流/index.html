常见的git工作流

概述

随着我们由svn迁移到git，我们也是不断的在探索一种安全可靠的工作流以提高工作效率。但又同时考虑到我们由中心集中式的思维转变为分布式版本系统的成本，所以目前效率云上采用的项目工作方式是–分支开发，分支发布。实际上这个和svn的思路转化不大，但感觉没有发挥出git分布式的好处。所以，介绍一下几种常见的git工作流，以方便大家更深层次的了解git的工作流模式。

git工作流–分支式工作流

在我们了解了集中分支工作流之后，下面我们来看这一种工作流—功能分支工作流。这种工作流主要有两个特点：
1，专门的功能在特定的分支上进行开发，逻辑更清晰，不干扰主代码库,主代码库只用于部署上线代码；
2，在功能开发完毕后，我们需要将代码合并到主代码库。这个时候我们就会新建一个pull request（merge reuqest）, 当然在这一步，我们可以进项code reciew，可以帮助团队对代码质量和规范进行比较严格的把控。

举个例子：

唐僧师傅交给悟空负责开发一个新的需求，这个时候，悟空接到开发任务，需要从最新的主分支上独立出来一个分支来工作。

当然这种情况下，悟空开始不断的提交修改。悟空辛苦开发了一天，该下班回家吃饭了。这个时候悟空需要把他当天写的代码提交到远程上。

这个时候，远程即有了feature-monkey的所对应的远程分支origin/feature-monkey。这种把代码推到远程的方式有效的解决了代码丢失的问题。比如你的电脑丢了、坏了、进水了，第二天你只需要拿来另外一台电脑，安上git，轻松继续开发。

几天后，悟空的这个功能开发完毕。feature-monkey这个分支需要合并主干(master)进行上线。这个时候，悟空需要创建一个merge request(pull request)，并将指向人指向经理唐僧师傅。唐僧师傅发现merge request，开始code review，发现悟空的代码没有问题允许合并到主分支。然后唐僧师傅进行合并，打tag进行上线操作。

当然，如果悟空同学的代码质量不佳或者有Bug，会被师傅打回重构代码。需要再进行一次编辑、暂存、提交。

同时，当一个分支被合并上线后，那么这个分支的声明周期就结束了，可以删除。

git工作流–集中式工作流

举个例子：

悟空、八戒和沙僧
实际上这种开发方式有点类似svn，开发人员基于master克隆项目进行本地开发，更改、提交都是本地的提交，这一点也是和svn类似。本地的提交是和远程的主干完全隔离的，等到合适的时机在进行远程的提交同步。

总结一下，集中式工作流就是维持一个中央库(master)，每个人克隆master，各自的提交到本地，在合适的时间节点push到master。

git工作流–gitflow工作流

Gitflow工作流为代码的开发和设计制作了一个较为严格的分支模型。这种方式虽然比功能分支的工作流稍微复杂一些，但这种方式恰恰对较大的项目管理有着很好的效果。

这种工作流的方式是将不同的分支分配了特定的角色，并且严格的说明了什么时候该操作，什么时候该交互。极大的利用了git分支的优点。

当然，我们还是要看图说话，先来说一说这些分支的具体作用：

Master分支： 只允许打tag以及版本发布，不允许提交代码。生命周期自然是无限长的稳定分支。
develop分支： 用来接收开发分支以及bugfix分支的代码，正常情况下很少从develop分支直接开发提交代码。生命周期也是无限长的不稳定分支。
release分支：功能开发完成后，用于准备生产版本。允许小错误的修复。
feature分支： 功能开发分支，用于一个项目或者功能的开发。
bugfix分支： 此分支用于解决开发中的bug，bug解决完成合并过后，此分支的生命周期就结束。
hotfix分支： 用于解决线上紧急Bug。

举个例子：

唐僧师傅给八戒分配了一个去化缘的任务，八戒接到任务，马上开整。

八戒的开发效率很快，几天之后，这个功能开发完毕了，需要合并到develop分支上。

但是谁却想八戒是个粗心的家伙，develop分支上出现了几个小问题

这个时候八戒需要基于develop创建bugfix-250分支来修复这几个小问题。修复完毕后合并到develop分支。

这个时候进入预生产环境环节，比如上一期的小版本是release-1.1,那么本期的也是一个小功能开发的话，八戒建立了release-1.2版本分支。

这个分支会存在的时间是在版本发布之前，当版本发布之后，这个分支的声明周期就不在了。有时候的小的错误修复也是在本分支进行的。当然是严禁在本分支添加大功能的。最后，八戒需要等待的是版本发布。

终于等到了这一天，八戒开发的要上线了。

当然，为了保证八戒所写的代码在master和develop的一致性，还需要合并一下develop。

总结：

其实不论是什么版本管理工具，只要有一种可靠的工作流方式或者是提升我们的工作效率，那就是适合我们的方式。git的工作流还有一些，比如交叉工作流等，这一些常玩github提过pull request的同学会深有体会。我只是介绍了几种我们日常开发的过程中常见的工作流，希望大家能够对git有一些更加深入的认识。
